当程序本身的性能瓶颈不依赖于数据结构共享就有保证性能的可能。说个真实的故事。我遇上过我司一位程序员，他做过一个非常有趣的实验，实验结果「证明」 Apache Spark 比 Apache Storm 在执行一个 streaming 计数逻辑时慢八千多倍，同时机器开支是 Storm 的十倍以上。当我看到这个结论时的脑袋里回荡着这样一句话：Σ(っ °Д °;)っ 我们不如去证明全球开源大数据社区全体成员集体精神失常了五年，我觉着还容易些。我抓了他的代码仔细读了一遍，发现他的逻辑是这样的：他在 Spark application 里维护了一个六小时的时间窗口，然后每两分钟窗口前移并将六小时时间窗口的所有数据过一遍，算出多少个 key 和每个 key 出现的次数，每一份数据输入都作为一个 RDD 保留在系统里。可能很多人都知道 Apache Spark 大量应用了函数式程序设计原则，其基本数据结构 RDD 就是不可变的，也就是说他的逻辑导致 Spark 在内存中维护了一个裸数据接近 40GB 的巨大队列，每两分钟就要遍历一遍。且不说反复重复计算的开销，Spark 做 checkpoint 的时候要把这个队列全部保存在 HDFS 里，这就得花掉大量的时间，这一笔一笔算起来不慢才有鬼。更何况要求 Spark 在内存中保存 40GB 的裸数据，算上数据副本和 Java 运行时的开销，弄出个十倍的机器开销简直是轻松愉快。那为什么 Storm 就很快呢？他的 Storm 逻辑直接在内存里维护了一个可写的 java.util.HashMap 哈希表……当然代价是这种原始的内存表扛不住机器掉线，可靠性堪忧。我们先别急着笑话他，他这么做是有原因的。因为一些我至今都不理解的业务需求，他的项目要求他每两分钟转储六个小时时间窗口内的全部裸数据。后来他也承认，其实他存了个心思用 checkpoint 来承担这个转储工作，因此花了大量时间研究怎么改造 checkpoint 让它来快速地存储裸数据上了，但 RDD 的不可变让他始终无法解决计算开销问题。——但熟悉 Spark 的朋友都知道 checkpoint 根本不是起这个作用的，改造 checkpoint 显然不是一个可行的思路。那么解决问题的方法是什么呢？Redis，我给他开了个 Redis 的队列和哈希表。计数的部分写哈希表，转储的部分写队列。机器用量重新回到了十分之一，两分钟处理完毕的性能要求也没问题。想转储？开一个异步进程遍历 Redis 队列就可以了，遍历 40 GB 数据怎么也用不了六个小时。所以啊，让上帝的归上帝，凯撒的归凯撒。

## 严格求值惰性求值以及纯函数式状态
非严格求值(惰性求值)是一项提升函数式编程效率和模块化的基础技术, 一个函数是非严格求值的意思是这个函数可以选择不对它的一个或者多个参数求值, 相反一个严格求值的函数总是对它的参数求值。在scala中我们可以通过接收某个未求值的参数来写非严格求值函数
``` scala
def if2[A](cond:Boolean, onTrue: ()=>A< onFalse ()=>A):A = if (cond) onTrue() else onFalse()

def maybeTwice(b:Boolean, i: =>Int) = if (b) i+i else 0

val x = maybeTwice(true, {println("hi"); 1+41})
```
()=>A 类型表示一个函数接受0个参数返回一个A类型 在 maybeTwice方法体中 i 被引用了两次 所以{}中的内容也会被执行两次(因为引用两次i的原因) 如果我们希望只求值一次 可以通过lazy关键字显式缓存这个值

``` scala
def maybeTwice(b:Boolean, i:=>Int)={
    lazy val j = i
    if (b) j+j else 0
}
val x = maybeTwice(true, {println("hi"); 1+41})
Scala> hi
Scala> x:48
```
对一个val声明的变量添加lazy修饰符, 将导致scala延迟对这个变量求值 直到第一次被使用的时候 同时也会缓存结果, 在后续引用的地方不会触发重复求值，这就是为什么只会看到打印一次hi

>严格求值的正式定义
如果对一个表达式的求值一直运行或抛出一个错误而非返回一个定义的值, 我们说这个表达式没有结束(terminate) 或者说它是evaluates to bottom 如果表达式f(x)对所有的evaluates to bottom的表达式是x 也是 evaluates to bottom 那么f是严格求值的

scala中非严格求值的函数接收的参数是传名参数(by name) 而非传值参数(by value)
